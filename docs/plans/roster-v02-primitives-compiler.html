ROSTER-V02-PRIMITIVES-COMPILER · RATIFIED DIRECTION

Make Roster a small public language for composing agents.

The chosen product is a curated filesystem library plus a small compiler and launcher. Roster defines reusable agent primitives and resolves them into inspectable bundles. Harnesses execute those bundles; Bitterblossom defines workflows and dispatches them; Powder records work.

Chosen design. Role composition is Roster's only semantic composition seam. An agent binds a role, primary model, Harness, and optional arguments. Resolution produces an immutable bundle; a Harness adapter may translate and launch it but may not change its meaning.

Outcome

A stranger can understand, compose, inspect, and dispatch a focused agent without adopting Misty Step's workstation or control-plane machinery.

Proof

Resolved bundle provenance, observable CLI fixtures, live Codex and OMP launches, unchanged user-owned Harness state, and the unchanged Rust gate.

Standard

Deep module, tiny surface. Opinionated primitives; provider-neutral mechanism; frontier-first prose; no hidden composition or override magic.

Stop if

The design grows a workflow engine, workstation manager, generic patch DSL, model taxonomy without eval evidence, or compatibility layer for v0 experiments.

01The product boundary

Sources

core, ~/.roster, client and eval roots. Canonical source-qualified IDs; shorthand only when unambiguous.

→

Role

One additive include list over skills, MCPs, guidance, and packs. This is the whole Roster-owned semantic composition.

→

Bundle

AGENTS.md, skills, MCP material, and YAML manifest with inclusion and adaptation provenance.

→

Harness

The agent's Harness binding selects one source-resolved adapter; it creates an ephemeral native projection, launches, and preserves exit/signal behavior.

Roster owns

  • Compositional primitives: skills, MCP declarations, and guidance.
  • Composition assets: packs and roles; launch assets: Harness adapters.
  • Multi-root discovery, validation, explanation, and provenance.
  • Bundle resolution and thin dispatch.
  • A concise first-party skill explaining the CLI.

Roster refuses

  • Tasks, claims, workflows, schedules, retries, and run supervision.
  • Workstation convergence, runtime health, secrets, auth, sessions, and caches.
  • HTTP/UI/MCP service faces without separately proven demand.
  • Provider-specific workflow semantics or OMP topology normalization.

03Opinion without coupling

Powder remains the public default, not a core schema field. Core roles may explicitly include a Powder work-management pack. A private client role includes the same general SDLC pack plus Linear instead. If the primitives differ, the role differs; the agent binding does not hide the difference.

Canonical primitive prose targets maximally capable models: concise, ambitious, explicit about why, and encouraging curiosity. Frontier models receive no elicitation overlay by default. Smaller-model scaffolding earns its way in through paired evals; it is not pre-authored as a combinatorial family of skill variants.

Any behaviorally different model variant becomes a newly identified primitive selected by the role. v0.2 adaptations may change articulation in resolved copies, but cannot include, replace, or semantically alter primitives. Harness-native arguments may select topology, budget, sandbox, or an external profile; Roster-owned definitions do not use them to smuggle in prompts, skills, or MCPs.

That supports generated routing, pointers to shared contracts, and deletion of repetition; it does not justify model-specific primitive forks.

08Oracle and review focus

Executable oracle

  1. Resolver fixtures prove canonical IDs, ambiguity failures, deduplication, and provenance.
  2. Bundle inspection proves generated routing matches included skills/MCPs and every adaptation is enumerated.
  3. Live Codex and OMP dispatches prove adapter feasibility without permanent workstation mutation.
  4. The unchanged formatting, lint, test, and Roster check gates remain green.

Adversarial review

  • Find any second semantic composition seam.
  • Find runtime/control-plane concerns masquerading as catalog/compiler work.
  • Find hidden source precedence or unrecorded adaptation.
  • Find model/provider claims that lack dated evidence.
  • Find valuable user-authored bytes the migration would silently discard.
Residual risk. The correct AGENTS generation strategy and capability-adaptation axis remain research questions. v0.2 must make both modular and observable without pretending they are settled.
