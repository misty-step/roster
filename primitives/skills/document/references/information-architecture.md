# Information Architecture — the taste

What separates world-class docs from an autogenerated wall of prose. This is the
judgment a model will not derive on its own; encode it, then let the repo's
shape choose specifics.

## Diátaxis is the spine

Four modes, different jobs, never blended on one page:

| Mode | Answers | For |
|---|---|---|
| Explanation | "why is it built this way?" | understanding — architecture, decisions, trade-offs, *lore* |
| Reference | "what exactly is X?" | lookup — APIs, schemas, config, CLI, exports |
| How-to | "how do I accomplish Y?" | a goal — add a feature, debug a flow, run locally |
| Tutorial | "teach me from zero" | a newcomer's first guided pass |

Most codebases are under-served on **Explanation** (the *why*) and over-served on
auto-generated **Reference** (the *what*, which the code already states). Weight
toward intent, flow, and decisions — the context that lives only in maintainers'
heads and the git history.

## Page taxonomy — types, not a fixed tree

Borrow types; let recon choose which exist and how they nest. A small library
needs five pages; a monorepo needs a section per service.

- **Overview / architecture** — the system's bones: major components, how they
  compose, how data and control flow. One good diagram beats three paragraphs.
- **Entry points** — where execution starts (main, handlers, CLI, jobs, cron).
- **Systems / modules** — one page per load-bearing subsystem; what it owns, its
  boundary, who calls it.
- **Data model** — schemas, key entities, persistence, migrations.
- **Interfaces** — public APIs, events, CLI surface, config/feature flags.
- **Build / test / deploy** — how to run it, prove it, and ship it.
- **Conventions** — the patterns a contributor must match (the non-inferable
  ones; skip what the code obviously shows).
- **Decisions / lore** — *why* the non-obvious choices were made; pulled from
  git history, ADRs, and code comments. The highest-value, lowest-supply pages.
- **Glossary / maintainers** — domain terms; who owns what.

## Diagrams that earn their place

A diagram must answer a question prose answers worse. Mermaid by default
(renders everywhere, diffs as text, parseable by the render oracle).

- **Architecture / container** — component composition (C4-ish). One per repo.
- **Sequence** — for the 2–4 flows that matter (request lifecycle, auth, the
  core job). Trace from ingress to persistence.
- **Data-flow / dependency** — module graph or data movement when coupling is
  the thing to understand.
- **State machine** — only for genuine state (orders, jobs, sessions).

No decorative diagrams. No diagram that just restates a list. Every node and edge
must be grounded in source the accuracy oracle can verify.

## What a world-class page looks like

- **Leads with the job-to-be-done**, not a definition dump.
- **Links to source lines** (`path:line`) so a reader jumps to ground truth, and
  the accuracy oracle has something to verify against.
- **Captures intent and flow**, never paraphrases function bodies — restated code
  is pure drift surface with no added understanding.
- **One diagram max** unless a second answers a distinct question.
- **Cross-links** to the pages a reader needs next; these cross-links are what
  the navigability oracle exercises.
- **States the why** for non-obvious choices, with a source (commit, ADR,
  comment) — unsourced rationale is a guess.

## Adapt to the system shape

- **CLI** — commands/flags as reference; one how-to per common task; a flow
  diagram of arg → action → output.
- **Service/API** — request lifecycle sequence; endpoint reference; data model;
  deployment topology.
- **Library/SDK** — public API reference; integration how-tos; a consumer-build
  quickstart; minimal internals unless contributors need them.
- **Monorepo** — a top-level map, then a section per package/service; resist one
  giant page.
